# Spec: documentation and roadmap sync to current state of the art

## Goal

Bring committed documentation in line with where the project actually
is: public repository, complementary positioning with respect to
existing tools, priority-ordered backlog of upcoming work, and
explicit architectural decisions about the future graphical interface
and programming-interface layer.

## Background

Between v1.0.0 (2026-04-27) and now, several decisions were made
off-tree:

- The repository was pushed public at `github.com/SpeederX/calibr`.
- Positioning was revised from "no tool of this kind exists" to a
  complementary frame against llmfit, llm-checker, llama-benchy,
  llama-bench, llama-sweep-bench, LocalMaxxing, and Bench360. Each
  addresses a different slice of the problem; calibr's slice is
  measuring launch-flag configurations on the user's own hardware.
- A new direction was set: download rotation as the next quick win,
  report-interface overhaul, on-hardware key-value-cache degradation
  test as the differentiating feature, a programming-interface
  refactor as the next landmark, multi-graphics-card planning as a
  long-shot landmark beyond that.
- Two interfaces, one engine: technical users on the command-line
  interface in this repository, less technical users on a future
  graphical interface in a separate `calibr-ui` repository, built on
  top of the v2.0.0 programming interface.

None of this is reflected in committed files. The `ROADMAP.md` Open
section lists only one code item. The `README.md` quickstart still
shows `your-username/calibr.git`. The "no tool exists" framing is in
prose. The architectural decisions about two surfaces and the future
programming-interface layer are nowhere on file.

## Behavior

### `README.md`

- The `git clone` URL in the quickstart points to
  `https://github.com/SpeederX/calibr.git`. Every occurrence of the
  string `your-username` in this file is replaced.
- The "Why not …?" prose drops any claim that no tool exists for this
  problem. The replacement framing acknowledges other tools and
  states calibr's complement: calibr measures launch-flag
  configurations on the user's own hardware; other tools estimate, or
  measure runtime parameters, or aggregate community data.
- The competitor table is introduced or rewritten with these rows,
  in this order:

  | Tool | Approach | Gap (relative to calibr) |
  |---|---|---|
  | llmfit | Pure estimation, hardcoded hardware constants | Does not measure on hardware; mixture-of-experts models treated as dense. TBD — pending hands-on test. |
  | llm-checker | Ollama-focused, deterministic scoring, `ai-run` subcommand measures tokens per second | No mixture-of-experts support; Ollama only; no launch-flag sweep. TBD — pending hands-on test. |
  | llama-benchy | Sweep over runtime parameters (prompt processing, token generation, depth, concurrency) | Sweeps runtime, not launch flags. TBD — pending hands-on test. |
  | llama-bench (built-in to llama.cpp) | Single-configuration benchmark | No sweep across launch flags. |
  | llama-sweep-bench | Sweep over performance parameters | Fork-specific to ik_llama; not applicable to mainline llama.cpp. |
  | LocalMaxxing | Community leaderboard aggregator | Depends on third-party submissions; no measurement on the user's own hardware. |
  | Bench360 | Academic benchmark framework | Not consumer-facing. |

- The "TBD — pending hands-on test" lines are placeholders that this
  spec does not resolve. A follow-up task refreshes those fields with
  measured findings.

### `ROADMAP.md`

The file is rewritten end-to-end. New shape:

- Header: project tagline, link to the priority-rationale design
  doc, single-screen-fit constraint preserved.
- `## Done` section: existing done list preserved verbatim.
- `## Open` section: items grouped by target version, in priority
  order from top to bottom.

  - `### v1.1.0 — quick wins`
    - Download rotation: download a model, benchmark it, delete it,
      move to the next. Working-set storage drops from roughly one
      hundred gigabytes to roughly twenty.
    - `localmaxxing-export` subcommand for distribution via the
      public leaderboard.
    - N-run with median (variance reduction; currently plus or minus
      five percent on `eval_tps` at N equals one).

  - `### v1.2.0 — report-interface overhaul`
    - Memory-tier reference lines on the scatter plot (video memory,
      video memory plus shared paging, video memory plus system
      memory).
    - Bar charts grouped by model, collapsed and expanded by the
      user.
    - Column-header tooltips and question-mark overlays explaining
      each metric.
    - All-results table sortable, searchable, with a group-by-model
      toggle.
    - Front-end test harness via Edge in headless mode.

  - `### v1.3.0 — key-value-cache degradation, measured on hardware`
    - Empirical quality test of key-value-cache quantization
      (eight-bit, four-bit, half-precision) on the user's own
      hardware. Open questions: metric, dataset, integration into
      the report.
    - Cross-reference llmfit's theoretical estimates against
      measured numbers; report the delta.

  - `### v1.4.0 and later — incremental`
    - Wattage tracking via the NVIDIA system management interface
      tool (`nvidia-smi`).
    - Efficiency metric (tokens per joule).
    - Accuracy task suite, fixed at five to ten representative tasks
      (general user-interface code, simple code generation).
    - Speculative-decoding planning.

  - `### v2.0.0 — programming-interface layer (landmark)`
    - Decide the form of the interface: local C library, local C++
      library, local web service, or another.
    - Refactor `calibr.ps1` into an engine library plus a
      command-line shell.
    - First reference client (likely the existing command-line
      interface itself).
    - This release opens the way for the future `calibr-ui`
      repository and possibly a web dashboard.

  - `### v3.0.0 — multi-graphics-card planning (landmark, long shot)`
    - Planning with `--tensor-split` across more than one card.
    - Per-card video-memory accounting and paging detection.

  - `### Cross-cutting (no version pin)`
    - Linux and macOS port: NVIDIA management library in place of the
      Windows Display Driver Model heuristic; shell rc files in place
      of the Windows user PATH; a `calibr` shell wrapper with no
      extension.
    - Android exploration (Termux plus ARM64 builds of llama.cpp).
    - Release automation (version bump, tag, changelog from commits).

- `## Architectural notes` section (new): records the explicit
  decisions, in plain language.

  - Two audiences, two interfaces. Technical users stay on the
    command-line interface offered by `calibr.ps1`. Less technical
    users will be served by a future graphical interface in its own
    repository, `calibr-ui`, built on top of this tool's
    programming interface. Same engine, two surfaces — not two
    versions of the tool, and not maintained in this repository.
  - A graphical interface or web dashboard lands at v2.0.0 or
    later, never before. Both depend on the programming interface
    shipped in v2.0.0.
  - The accuracy task suite stays narrow: five to ten representative
    tasks. It is not a general evaluation framework, and avoiding
    that scope creep is the explicit choice.

- Final competitor-table content; the stub rows are intentional.
- Outreach content of any kind (social posts, direct messages,
  community announcements).
- Any change to engine behavior, configuration schema, samples, or
  test suite. The only engine touch is the version constant bump,
  covered by the plan.
