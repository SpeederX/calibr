# Plan: v1.0.1 — post-v1 documentation and methodology sync

## Context

Two and a half weeks after v1.0.0, the project's state of the art on
disk has drifted from the actual state of the art:

- The repository is now public at `github.com/SpeederX/calibr`. The
  "while offline, breaking minor bumps allowed" paragraph in
  `architecture/README.md` no longer applies.
- Strategic decisions made off-tree need to land in committed files:
  drop the "no tool of this kind exists" framing, position calibr in
  a complementary frame against existing tools, reorder the open
  work, add a graphical-interface and programming-interface direction
  that did not exist before.
- `ROADMAP.md` only reflects work as of the v1.0.0 commit. The new
  priorities (download rotation, leaderboard export, median-of-N runs,
  report-interface overhaul, key-value-cache degradation differentiator,
  future programming-interface layer, future multi-graphics-card
  planning) are nowhere on file.
- `memories/memory-2.md` is the most recent snapshot, dated
  2026-04-27. Its "next planned work" pointer (a report-interface
  overhaul as v1.1.0) is superseded.

This plan brings the committed state in line with the current state.
No engine behavior changes; documentation and methodology only.
Bump: v1.0.1 (patch).

Prerequisite, separate branch. An in-flight branch
`feat/v1.1-ux-overhaul` holds docs (spec + plan) for the
report-interface overhaul, originally pinned to v1.1.0. Under the
new ordering decided for this sync, that work targets v1.2.0.
Before this plan executes, a one-commit relabel on that branch
updates the internal v1.1.0 references to v1.2.0, renames the plan
file accordingly via `git mv`, and renames the branch to
`feat/v1.2-ux-overhaul`. The branch stays unmerged. This plan and
its commits do not touch those files.

Implements:

- [spec/v1.0.1-docs-and-roadmap-sync.md](../spec/v1.0.1-docs-and-roadmap-sync.md)
- [spec/v1.0.1-strict-semver.md](../spec/v1.0.1-strict-semver.md)

Adds:

## Approach

Single branch `chore/v1.0.1-post-v1-sync` off `dev`. Atomic commits
in the order below; each one is reviewable on its own and leaves the
repository in a coherent state.

### Commit order

1. **`docs: design rationale for the priority-ordered backlog`**
   New file `architecture/design/roadmap-priorities.md`. Standard
   design-doc shape (Why / Pros / Cons / Takeaway). Explains why the
   roadmap is a priority-ordered backlog grouped by target version
   rather than a date-ordered timeline, and why landmark features
   land at major bumps.

2. **`docs: specs and plan for v1.0.1`**
   This file plus `spec/v1.0.1-docs-and-roadmap-sync.md` and
   `spec/v1.0.1-strict-semver.md`. Update `spec/README.md` and
   `plans/README.md` indices.

3. **`docs: restore strict semantic versioning`**
   Implements `spec/v1.0.1-strict-semver.md`. Remove the
   "Pre-public flexibility" paragraph from `architecture/README.md`
   and replace it with a single sentence confirming strict semantic
   versioning. Append a dated note in `memories/memory-2.md`
   recording that the allowance applied to v1.0.0 only.

4. **`docs: rewrite README positioning and seed competitor stubs`**
   Implements the README portion of
   `spec/v1.0.1-docs-and-roadmap-sync.md`. Fix the
   `your-username/calibr.git` placeholder. Drop the "no tool exists
   for this" framing. Add or rewrite the competitor table with rows
   for llmfit, llm-checker, llama-benchy, llama-bench,
   llama-sweep-bench, LocalMaxxing, Bench360 — columns: name,
   approach, gap. Each gap field set to "TBD — pending hands-on
   test" where the maintainer has not yet validated it.

5. **`docs: restructure ROADMAP as priority-ordered backlog`**
   Implements the ROADMAP portion of
   `spec/v1.0.1-docs-and-roadmap-sync.md`. Whole file rewritten.
   New shape: Done section preserved verbatim; Open section grouped
   by target version (v1.1.0 through v3.0.0); a cross-cutting
   section for items without a version pin; an Architectural notes
   section recording the three explicit decisions (two interfaces,
   graphical interface at or after v2.0.0, narrow accuracy task
   suite).

6. **`docs: memory-3 snapshot — public, strict semver, new backlog`**
   New file `memories/memory-3.md`. Captures: public repository
   status, strict semantic versioning in effect, the priority-
   ordered backlog model, the two-surfaces architecture (command-
   line interface in this repository, future `calibr-ui` repository
   on top of the v2.0.0 programming interface), pointers in reading
   order, caveats.

7. **`chore: bump version to v1.0.1`**
   Update the version constant in `calibr.ps1` from `1.0.0` to
   `1.0.1`. Add a `[x] **v1.0.1**: documentation and methodology
   sync` line at the bottom of the Done section in `ROADMAP.md`.

## Risk and trade-offs

- **Competitor stubs are placeholders.** Gap fields read "TBD —
  pending hands-on test" until a follow-up does the testing. The
  trade-off is shipping the corrected positioning now rather than
  waiting on the testing task. Stubs are honest about their state.
- **The new backlog format is opinionated.** A reader has to accept
  that "next" means "top of the highest-priority open section." The
  design doc explains why; if the convention is wrong, the doc is
  where the discussion happens, not the roadmap itself.
- **memory-2.md is left as-is, with one dated note.** Its "next
  planned work" pointer becomes historically wrong on the day
  memory-3.md lands. This is consistent with the memory-snapshot
  convention: older snapshots are historical, not authoritative.
  The dated note in memory-2 is the minimum needed to record that
  the relaxed-versioning allowance was time-bounded.

## Out of scope for this plan

- Hands-on competitor testing and full competitor-table refresh.
- Any engine work toward v1.1.0 (download rotation, leaderboard
  export, median-of-N runs).
- Release automation (version bump, tag, changelog generation).
- Any outreach activity. Outreach is not project content and never
  lands in committed files.
