# Plan: v1.1.0 — quick wins (download rotation, localmaxxing
# export, N-run median)

## Context

The first post-v1.0.1 release. Three orthogonal "quick wins" sit
at the top of v1.1.0 in the new priority-ordered backlog:

- **Download rotation** so `-DownloadSamples` is usable on small
  solid-state drives (working set drops from roughly one hundred
  gigabytes to roughly twenty).
- **localmaxxing-export** so calibr's measured numbers feed back
  into the community leaderboard at localmaxxing.com, where users
  can compare across hardware.
- **N-run median** so the ±5 % variance on `eval_tps` does not
  silently mask small regressions in subsequent releases.

Each item is independently shippable; the plan bundles them
because they share v1.1.0 as a release vehicle. Strict semantic
versioning is in effect (per v1.0.1): every behavior under each
new flag is additive or opt-in, so a MINOR bump applies.

Bump: v1.1.0 (minor, backwards-compatible).

Implements:

- [spec/download-rotation.md](../spec/download-rotation.md)
- [spec/localmaxxing-export.md](../spec/localmaxxing-export.md)
- [spec/n-run-median.md](../spec/n-run-median.md)

Adds:

- `data/results/*.json` gains a `runs` array and a
  `localmaxxing_export` sub-object (per-result).
- `data/catalog.json` gains `downloaded_by_calibr` and
  `hf_model_id` fields per catalog entry.
- `config.default.json` gains `bench.runs_per_config` and
  `localmaxxing.auto_submit`.

## Approach

Single branch `feat/v1.1-quick-wins` off `dev` (after v1.0.1 is on
`master`). Atomic commits in the order below; each one is
reviewable on its own and leaves the repository in a coherent
state.

### Commit order

1. **`docs: specs and plan for v1.1.0`**
   Three specs plus this file. Update `spec/README.md` and
   `plans/README.md` indices.

2. **`feat(bench): N-run median with per-run capture`**
   Implements `spec/n-run-median.md`. Adds `bench.runs_per_config`
   to `config.default.json` (default 3). Adds `-Runs` flag to
   `bench` and `all`. Refactors `Invoke-OneBench` so the inner
   warmup-then-bench loop runs N times; collects the N raw
   per-run records into a `runs` array; computes medians for the
   varying metrics and writes them as the top-level result. Adds
   a median helper plus tests for it (N=1, odd-N, even-N). Adds
   a `Config.Tests.ps1` row for the new field.

3. **`feat(get-sample-models): tag downloaded files in the catalog`**
   Prerequisite for rotation: catalog entries for files calibr
   itself downloads gain `downloaded_by_calibr: true`. User-owned
   files discovered by `discover` are left untagged (treated as
   `false`). One unit test for the tagging path.

4. **`feat(all): -Rotate flag for -DownloadSamples`**
   Implements `spec/download-rotation.md`. Adds the `-Rotate`
   flag. After each model's configs complete, if all configs
   succeeded AND the catalog entry has `downloaded_by_calibr:
   true`, delete the `.gguf` and any auto-paired `mmproj-*.gguf`
   from disk. Catalog entry is preserved. Usage error if
   `-Rotate` is passed without `-DownloadSamples`.

5. **`feat(get-sample-models): record hf_model_id at download time`**
   Prerequisite for export of sample-set models: at download
   time, the Hugging Face model id is derived from the URL in
   `samples.json` and written to the catalog entry as
   `hf_model_id`. Backfills are not attempted (existing catalog
   entries without the field stay null until next download or
   until the user supplies them via the export prompt).

6. **`feat: localmaxxing-export subcommand`**
   Implements the bulk of `spec/localmaxxing-export.md`. New
   `Invoke-LocalmaxxingExport` function. Reads
   `$env:CALIBR_LOCALMAXXING_API_KEY`; if missing, prints the
   one-line warning and exits zero. Maps each unsubmitted
   `data/results/*.json` to the documented localmaxxing schema.
   POSTs to `/api/benchmarks`; records outcome in
   `localmaxxing_export` sub-object. Prompts for missing
   `hf_model_id` on user-owned models (interactive) and persists
   to the catalog. `-Force` overrides "already submitted" skip.

7. **`feat(all): auto-submit opt-in prompt`**
   First-run prompt during `calibr all` when
   `localmaxxing.auto_submit` is unset. Stores the answer in
   `config.json`. With `auto_submit: true`, `all` invokes the
   export step after `report`. With `-NonInteractive`, the
   prompt is skipped and `auto_submit` defaults to false.

8. **`feat(report): localmaxxing submissions panel`**
   `report.template.html` gains a "localmaxxing submissions"
   section: two summary numbers (submitted, failed) and an
   expandable per-result block for failures (test id, http code,
   error message). Static when there is nothing to show. Report
   smoke tests assert the section's structural elements exist.

9. **`docs: ROADMAP done + memory-4 post-v1.1 snapshot`**
   Mark the three v1.1.0 items shipped in `ROADMAP.md`. Add
   `memories/memory-4.md` with the post-v1.1.0 state (new flags
   in use, expected variance numbers from a follow-up
   measurement pass, the same reading-order pointers updated).

10. **`chore: bump version to v1.1.0`**
    Update the project version source-of-truth (see Risk and
    trade-offs: the v1.0.1 cycle deferred surfacing the version
    in `calibr.ps1`; v1.1.0 inherits that deferral). Add
    `[x] **v1.1.0**: download rotation, localmaxxing-export,
    N-run median` to the ROADMAP Done section.

## Risk and trade-offs

- **Tripling bench wall-time by default.** `runs_per_config: 3` is
  a 3× cost vs the current single run. Mitigation: `-Runs 1`
  reproduces the old behavior exactly; the release note flags
  the change and points at the flag.
- **localmaxxing API contract is external.** A future change to
  the schema at localmaxxing breaks our export until calibr is
  patched. We pin to the api-docs as of v1.1.0; the integration is
  isolated in one function (`Invoke-LocalmaxxingExport`) so the
  blast radius is bounded.
- **Rotation depends on tag accuracy.** A user-owned `.gguf` that
  gets re-recorded as `downloaded_by_calibr: true` (because of a
  bug) would be deletable. Mitigation: tagging only happens at
  the `get-sample-models` download path, never in `discover`. A
  follow-up audit step (a Helpers test) verifies the boundary.
- **HF-id prompt during export is interactive.** Users running
  export in CI hit the prompt. Mitigation: `-NonInteractive`
  skips the prompt and excludes those results from this call;
  the user can pre-populate the catalog with the ids in a
  separate `config set` (or by editing the catalog by hand) for
  CI flows. Documented in the spec.
- **Submission status leaks across re-benches.** A re-bench
  re-writes the result file from scratch and the
  `localmaxxing_export` sub-object resets to absent. The next
  export attempts the submission again. That is the intended
  behavior (numbers changed, submit again) but is worth a release
  note.
- **Version surfacing still deferred.** Step 10 bumps the tag and
  the ROADMAP Done line but does not touch `calibr.ps1` (no
  version constant exists; introducing one is on the v1.4.0+
  backlog). v1.1.0 inherits the v1.0.1 deferral; the tag remains
  the source of truth.

## Out of scope for this plan

  Independent follow-up; v1.1.0 ships the export plumbing but does
  not depend on any competitor test completing.

- Adaptive N or spread metrics beyond the median.
- UI changes that surface the raw per-run values in the report.
  Audit users read the JSON.

- Surfacing the project version in `calibr.ps1`. Tracked as a
  v1.4.0+ backlog item.
- Release automation (version bump, tag, changelog). v1.1.0 still
  uses the manual release flow.
