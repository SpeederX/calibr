Throughput & memory

?

A. Fully in VRAM: model weights, KV cache and compute buffers fit in GPU memory. This is the clean case; eval tokens/s usually stays stable.

B. Near the cliff: VRAM is almost saturated. Small changes in context, KV type, background apps, or MoE routing can push the run over the edge.

C. Spill / paging: Windows WDDM starts using shared system RAM. The model may still load, but token/s can drop hard because data crosses CPU/RAM/PCIe instead of staying in VRAM.

Baseline: Baseline is determined empirically by measuring system VRAM before starting the benchmark and before each configuration. Formula: baseline % = VRAM used before run / total VRAM × 100.

Example: 1500 MiB used / 8192 MiB total = 18.3% already used by the system. A warning appears when the configured threshold (for example 5%, 10%, or 15%) is lower than or equal to that percentage.

Run VRAM: The memory charts subtract that baseline from the system-level peak: estimated run VRAM = total peak - baseline. Apps, 3D processes, browsers, overlays, or background work can still change the peak and affect the benchmark.

MoE note: MoE can show the same pressure when experts do not fit cleanly. CPU expert offload may be intentional, but it is still a latency tradeoff.

Latency fields: TTFR is first streamed server response; e2e TTFT is first generated content/reasoning token; prompt ms is llama.cpp prefill time; request ms is the measured request wall time.
